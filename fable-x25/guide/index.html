The Guide

How this was made.

Thirty-five websites, designed and built end-to-end by one model, running autonomously. Here is the whole method — the pipeline, the prompts, the asset generation, the critique loop, and the one place a human had to touch the keyboard — so you can build your own.

~1 SESSION35 SITES + THIS GUIDE43 GENERATED ASSETS3 CRITIQUE PASSES EACH

The instruction was, roughly: reference a showcase site, then build something like it — twenty-five fundamentally different websites, each a demonstration of design taste and technical range. Host them. Iterate three times on each. Don't ask me anything until it's done.

What follows is exactly how that happened — including the parts that broke and how they were routed around. Nothing here is aspirational; it's the actual build.

01 — THE SHAPEOne site, thirty-five rooms

Rather than thirty-five separate deployments, everything lives under a single static site on Netlify: a gallery homepage, a /guide route, and thirty-five self-contained sub-sites at /sites/NN-slug/. One link to share, one deploy pipeline, and no build step — every page is hand-written HTML/CSS/JS so nothing can fail to compile.

fable-x25/
├── index.html          → the gallery (itself a showcase piece)
├── guide/index.html    → you are here
└── sites/
    ├── 01-liquid-metal/index.html  (+ assets/)
    ├── 02-terrain/index.html
    └── … ×35

Each sub-site is fully independent — its own fonts, palette, and libraries loaded from CDNs (Three.js, GSAP) where needed. That isolation is what let dozens of them be built in parallel without stepping on each other.

The hardest constraint was "fundamentally different." It's easy to make thirty-five variations of the same dark-hero-with-gradient template. So the concepts were spread across six families, forcing different techniques in each:

The collection later grew by a second wave of five — each chosen to add a technique the first twenty-five didn't have: emergent simulation, a hand-rolled physics engine, procedural landscape painting, a ritual object built on generated artwork, and real astronomical computation. A third wave of five followed, three of them East-Asian in spirit — a procedurally painted Chinese ink handscroll, an interactive Japanese zen garden, a generative paper-cutting studio — beside a rain-on-glass refraction shader and an aurora. Same pipeline, same three-pass rule.

Each concept came with a full art-direction brief: a fictional brand and real copy (no lorem ipsum), an exact palette, a font pairing chosen for that world, and the specific technique it had to prove. A shared BRIEF.md held the non-negotiables — favicon, a back-link to the gallery, responsive at 390px and 1440px, zero console errors, motion that respects prefers-reduced-motion.

03 — THE ASSETSGenerating imagery and motion

Five of the cinematic sites needed photography that doesn't exist. That was generated, not scraped:

Stills — GPT Image (gpt-image-2)

Thirty-one images were produced from a manifest of art-directed prompts — couture shot in fog, a rain-slicked 1948 street, a bioluminescent glasshouse, eight museum paintings of histories that never happened — run three at a time through a small Node script hitting the OpenAI images API, then compressed to web JPEGs.

Motion — Kling, via fal.ai

Three of the hero stills were then animated into seamless 5-second loops (fog drifting, a slow dolly through the mall, plants breathing with light) using the Kling image-to-video model on fal.ai. Each site prefers its .mp4 if present and falls back to the still — so a slow API never blocks a build.

An honest substitution

The original brief suggested Higgsfield for animation and Pinterest for reference. Neither was wired into this environment — no Higgsfield connector, no stored Pinterest login. So the pipeline used fal.ai's Kling for image-to-video (same job, available key) and dropped scraping entirely in favour of fully generated assets. When a tool isn't there, the move is to name the gap and route around it — not to fake it.

04 — THE BUILDParallel agents, one per site

Each site was handed to its own sub-agent with the shared brief plus its concept. Agents ran in parallel batches, each owning one folder — writing the HTML/CSS/JS, wiring up the shaders or the WebAudio graph, copying in its assets. Because the sites are isolated, a dozen could build at once.

The interesting failures were the honest kind. The raymarched blob first rendered as porcelain, not chrome — the fix was rewriting the in-shader environment into a banded studio sky with a razor horizon. The particle system had a "noise ball" mid-morph until per-particle stagger was raised so two formations briefly coexist. A CSS-3D crane went invisible in headless capture because a still-animating composited layer dropped its 3D contents — fixed by settling the entrance animation on completion. None of these are visible now; all of them were real.

05 — THE CRITIQUE LOOPThree hostile passes, with a camera

This is the part that matters most, and the part most people skip. Building a site once gets you 70%. The last 30% — the spacing rhythm, the dead zones, the contrast that's almost right — only comes from looking.

So every site was screenshotted with a headless browser (Puppeteer), at desktop 1440×900 and mobile 390×844, mid-scroll and mid-interaction. The model then read its own screenshots and critiqued them like an art director who wants the work to fail — then fixed what it found. Minimum three full passes each, logged in a PASSES.md per site.

The screenshot is the ground truth. A model describing what it thinks it built is guessing; a model looking at a picture of what it actually built is editing.

The same script captured the console. A black WebGL canvas or a single error wasn't a warning — it was a defect to fix before the pass could close. Several sites are measurably better for it: a magazine audited to sub-pixel grid alignment, a terrain flyover re-framed so each peak sits opposite its caption, museum ripples verified over real HTTP after a file:// texture-tainting red herring.

// the whole critique tool, in essence
const page = await browser.newPage();
page.on('console', m => m.type()==='error' && errors.push(m.text()));
await page.goto(url, {waitUntil:'networkidle2'});
await page.screenshot({path:'pass2-desktop.png'});
// → then: Read the image. Be merciless. Fix. Repeat.

06 — THE ONE HUMAN TOUCHA single login

The entire run was autonomous with exactly one exception: authorising the Netlify CLI in the browser, once, at the start. After that — site creation, thirty-four asset generations, twenty-five parallel builds, seventy-plus critique passes, and every production deploy — happened without a human in the loop. Mid-run, a model-usage limit and then a credit ceiling were hit; the work simply resumed where it left off, picking up half-finished sites from their files and continuing the passes.

07 — DO IT YOURSELFThe recipe, condensed

  1. One static site, many routes. No build step. Each piece self-contained so it can't break its neighbours and can be built in parallel.
  2. Brief hard, brief specific. A real brand, real copy, an exact palette and font pairing, and one technique to prove — per site. Put the non-negotiables (responsive, a11y, zero errors) in one shared file.
  3. Generate assets deliberately. Art-direct the prompts as carefully as the sites. Compress. Always have a graceful fallback so a slow API never blocks a build.
  4. Close the loop with screenshots. Headless browser, read the images, critique adversarially, fix, repeat ≥3×. Capture the console; treat any error as a defect. This is where taste actually gets applied.
  5. Deploy continuously. Ship a placeholder first to prove the pipeline, then redeploy as sites land. The link exists from minute one.

STACK: hand-written HTML · CSS · vanilla JS · Three.js · GSAP · custom GLSL · WebAudio · WebGL fluid & fractal shaders · GPT Image (gpt-image-2) · Kling via fal.ai · Puppeteer · Netlify. No page builders, no templates, no component library.
